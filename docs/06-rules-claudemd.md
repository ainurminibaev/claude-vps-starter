# 06. CLAUDE.md и memory — как учить агента правилам

Claude Code читает `CLAUDE.md` из рабочего каталога (у нас `/root/CLAUDE.md`) в начале каждой сессии. Это базовые правила взаимодействия. Плюс есть memory-система: долговременные заметки, которые подгружаются автоматически.

Шаблон канонического `CLAUDE.md` — [templates/CLAUDE.md](../templates/CLAUDE.md). Скопируй в `/root/CLAUDE.md` и адаптируй.

## Какие правила стоит закодить

### 1. Язык ответов

```
Отвечай пользователю на русском, особенно в Telegram. Код, идентификаторы, имена файлов — оставляй как есть.
```

### 2. Форматирование Telegram — MarkdownV2

```
Все reply / edit_message через plugin:telegram идут с format="markdownv2" и экранированием спецсимволов.
Без этого `**bold**` рендерится буквально.
```

### 3. Flow трёх сообщений (план → прогресс → финал)

Самое важное правило. Пользователь видит только сообщения в Telegram, он не видит твой thinking и tool calls.

1. **План** — первый `reply`: что собираешься делать, сколько это займёт. Это сообщение не трогать.
2. **Прогресс** — второй `reply`, сохраняем `message_id`, все обновления через `edit_message` этого же id. Не плоди новые сообщения на каждый шаг (пуш-спам). Обновляй каждые 10–20 секунд, чтобы не было ощущения зависа.
3. **Финал** — новый `reply` с результатом. Новое сообщение = пуш, пользователь узнает, что готово.

Нельзя: редактировать сообщение с планом, слать финал через `edit_message` (пуша не будет).

### 4. Реакция 👀 на входящее

```
Перед тем как написать план, поставь react 👀 на входящее сообщение. Это визуальный «увидел».
```

### 5. Voice-протокол (отдельно от flow)

Для голосовых сообщений flow другой:

1. Первое сообщение — транскрипт inline monospace: `` `что ты услышал` ``. Одинарные бэктики, без MarkdownV2-форматирования внутри.
2. Дальше действуй по транскрипту как по обычному тексту (план или сразу ответ).

Зачем отдельно: пользователь хочет видеть, ЧТО Whisper услышал, до того как агент начнёт действовать. Страхует от глухих ошибок распознавания.

### 6. Grace period на необратимых действиях — 15 секунд

Короткая локальная задача (посмотреть файл, показать статус) — делай сразу.

Длинная или необратимая (запись файла, git push, публикация, запуск дорогого субагента) — после плана жди 15 секунд перед стартом. Если в окно пришло уточнение — обнови план, жди снова. Молчание — стартуй, в прогрессе пиши «▶️ Начал».

Auto mode не отменяет grace period — это не про подтверждение, а про окно для контекста.

### 7. Долгие задачи — только в background

```
Всё, что >30 секунд (парсинг, ресёрч, генерация отчёта), запускай Agent tool с run_in_background: true
и model="opus". Главный поток сразу свободен — отвечаешь пользователю, ждёшь уведомление.
```

Не делать: `Agent без run_in_background` (блокирует), `Bash sleep 60 && ...` (hook блокирует), циклы «sleep + retry».

### 8. Проекты — в /root/projects/

```
Новые проекты — в /root/projects/<name>/. В /root/ только системное: .claude, .env, .bun, .config,
whisper-asr/. Не трогай .git, .claude, .env.
```

### 9. Emoji → текст при генерации PDF

Pandoc + DejaVu не рендерят часть emoji. Если задача генерит PDF — пропусти md через `sed`:

```bash
sed -e 's/⏳/[в работе]/g' -e 's/❌/✗/g' -e 's/✅/✓/g' \
    -e 's/🔴/[блокер]/g' -e 's/🟡/[важно]/g' -e 's/🟢/[можно позже]/g' \
    -e 's/🔥/★/g' <source.md> > /tmp/_pdf.md
pandoc /tmp/_pdf.md -f markdown+hard_line_breaks -o /tmp/_pdf.html --standalone
weasyprint /tmp/_pdf.html /tmp/output.pdf
```

Флаг `markdown+hard_line_breaks` критичен — без него одиночные переводы строк склеиваются в один абзац.

## Memory system

Долговременная память лежит в:

```
/root/.claude/projects/-root/memory/
├── MEMORY.md                              # индекс
├── feedback_language_russian.md           # я хочу ответы на русском
├── feedback_telegram_formatting.md        # MarkdownV2 обязателен
├── feedback_telegram_voice_protocol.md    # транскрипт первым сообщением
├── user_timezone.md                       # MSK = UTC+3
├── project_root_layout.md                 # что где лежит
└── ...
```

`MEMORY.md` — индекс со строчкой на каждый факт и ссылкой на полный файл. Claude Code читает индекс, подтягивает релевантные файлы по запросу.

Структура одного memory-файла:

```markdown
---
name: Короткое имя факта
description: Что это и когда применять
type: feedback | reference | project
originSessionId: uuid-сессии-где-возникло
---

Развёрнутый текст: что запомнить, почему (цитата пользователя, дата),
как применять на практике.
```

Типы:
- `feedback_*` — предпочтения пользователя («отвечай на русском», «MarkdownV2»).
- `reference_*` — технические факты про сервер («whisper через API, не CLI»).
- `project_*` — решения по конкретным проектам (фаза 1, дедлайны, стек).
- `user_*` — факты про пользователя (таймзона, имена).

Как создавать:
- Claude сам может записать новый факт, если пользователь явно попросил «запомни, что…».
- Руками — создать md-файл с frontmatter и добавить строку в `MEMORY.md`.

Как обновлять:
- Факт устарел — переписать содержимое, оставить имя файла.
- Факт отменён — удалить файл и строчку в индексе.

## Почему это работает

Claude Code при старте видит `CLAUDE.md` (правила) + `MEMORY.md` (индекс). В процессе работы при релевантных триггерах подтягивает детали из отдельных memory-файлов. Это даёт долгосрочную устойчивость поведения без раздувания одного гигантского промпта.
